## 4. 프로젝트 상세

### Slide 5. Project Overview

**프로젝트명**  
Jmarket - 중고거래 및 경매 기반 마켓 플랫폼

**핵심 문장**  
중고거래 과정에서 발생하는 상품 탐색, 거래 요청, 경매 입찰, 채팅, 마일리지 정산, 관리자 운영 문제를 해결하기 위해 통합 마켓 플랫폼을 개발했습니다.

**개발 배경**

기존 중고거래 서비스는 상품 거래, 경매, 채팅, 신고, 정산, 관리자 운영 기능이 각각 분리되어 있어 전체 거래 흐름을 한 번에 관리하기 어렵습니다. Jmarket은 일반 사용자와 관리자의 관점을 모두 고려해 거래 전후 과정을 하나의 서비스 흐름으로 구성하는 것을 목표로 했습니다.

**해결하려는 문제**

- 상품 등록 이후 거래 요청, 채팅, 거래 완료까지의 흐름 관리
- 경매 입찰 시 최고가, 입찰자, 마일리지 예약 상태 정합성 유지
- 신고, 출금, 회원 제재 등 관리자 운영 기능의 분리
- 사용자 권한에 따라 메뉴와 버튼, API 접근을 다르게 제어

## 4-3. 아키텍처

### Slide 13. System Architecture

**전체 구조**

```text
React Frontend
  - 사용자 화면
  - 관리자 화면
  - 채팅 팝업
        |
        | REST API / WebSocket
        v
Spring Boot Backend
  - Auth / Product / Auction / Trade
  - Chat / Mileage / Report / Admin
        |
        +---- MySQL
        |       - 사용자, 상품, 경매, 거래, 마일리지, 신고 데이터
        |
        +---- Redis
                - 이메일 인증, 캐시성 데이터
```

**시스템 흐름**

1. 사용자가 React 화면에서 상품/경매/거래 요청
2. Spring Boot API에서 인증/인가 및 도메인 상태 검증
3. MySQL에 핵심 데이터 저장
4. Redis를 이메일 인증 및 캐시성 기능에 활용
5. 채팅은 WebSocket/STOMP로 실시간 메시지 송수신

## 4-5. 핵심 로직

### Slide 16. Core Logic 1 - Auction Bid / Mileage Reservation

**로직 이름**  
경매 입찰 및 마일리지 예약 처리

**구현 이유**

경매 입찰은 단순히 입찰 금액만 저장하는 기능이 아니라 현재 최고가, 입찰자 권한, 경매 상태, 마일리지 잔액, 이전 입찰자의 예약 해제를 함께 처리해야 합니다.

**문제 상황**

여러 사용자가 같은 경매에 입찰할 경우 최고 입찰자와 마일리지 예약 상태가 불일치할 수 있습니다.

**해결 방법**

- 입찰 요청 시 경매 상태와 마감 여부 검증
- 현재 최고가 기준 최소 입찰 금액 검증
- 본인 상품 입찰 및 현재 최고 입찰자 재입찰 차단
- 신규 최고 입찰자의 마일리지 예약
- 이전 최고 입찰자의 예약 마일리지 해제
- 경매 마감 시 최종 낙찰자 기준 정산 처리

**결과**

경매 최고가, 최고 입찰자, 마일리지 예약 금액이 일관되게 유지되도록 구현했습니다.

---

### Slide 17. Core Logic 2 - Role Based Access Control

**로직 이름**  
권한 기반 메뉴 및 API 접근 제어

**구현 이유**

프론트에서 버튼이나 메뉴를 숨기는 것만으로는 보안이 보장되지 않습니다. 사용자가 직접 관리자 API를 호출할 수 있기 때문에 서버에서 권한 검증이 필요했습니다.

**해결 방법**

- 사용자 역할에 따라 프론트 메뉴/버튼 노출 제어
- 관리자 API는 Spring Security 설정에서 서버 접근 제한
- 관리자 페이지 진입 시 권한 없는 사용자는 접근 차단
- 신고/출금/회원 제재 같은 관리자 액션은 서버에서 재검증

**결과**

UI 편의성과 서버 보안을 분리하여 권한 없는 사용자의 관리자 기능 접근을 방지했습니다.

---

### Slide 18. Core Logic 3 - Image Upload Validation

**로직 이름**  
이미지 업로드 서버 검증

**구현 이유**

프론트 검증만으로는 확장자를 위장한 파일 업로드를 막기 어렵기 때문에 서버에서 실제 파일 형식을 검증해야 했습니다.

**해결 방법**

- 최대 업로드 개수 제한
- 파일당 용량 제한
- 허용 확장자 검증
- Content-Type 검증
- 확장자와 Content-Type 일치 여부 확인
- JPEG/PNG/GIF/WEBP 파일 시그니처 검사
- 저장 경로 검증

**결과**

정상 이미지 업로드, 위장 이미지 차단, 미지원 확장자 차단 테스트를 작성하여 업로드 안정성을 높였습니다.

### Slide 19. Core Logic 4 - React Data Loading Flow

**로직 이름**  
React Hook 의존성 및 데이터 로딩 안정화

**구현 이유**

`useEffect` 의존성 경고를 단순히 자동 수정하면 API 중복 호출, 검색 필터 초기화, 결제 리다이렉트 재처리 같은 문제가 발생할 수 있었습니다.

**해결 방법**

- 조회 함수는 `useCallback`으로 안정화
- 검색/필터 화면은 URL query 기반으로 상태 복원
- 조회 함수가 컴포넌트 상태 기본값에 의존하지 않도록 필터를 명시적으로 전달
- 결제 리다이렉트 처리 후 query 제거 로직 보정
- 경매 마감 타이머는 auction 객체 전체가 아닌 필요한 값만 의존

**결과**

React Hook dependency warning 17개를 제거했고, `npm run lint`, `npm run build` 기준 경고 없이 빌드 가능하도록 개선했습니다.

## 4-6. 트러블슈팅 / 성능 개선

### Slide 20. Troubleshooting 1 - Auction Consistency

**문제 상황**

경매 상품에 여러 사용자가 입찰할 때 최고 입찰자, 현재 최고가, 마일리지 예약 상태가 서로 불일치할 가능성이 있었습니다. 예를 들어 최고가는 갱신되었지만 이전 입찰자의 예약 마일리지가 해제되지 않거나, 동일한 조건의 입찰이 중복 처리될 수 있었습니다.

**원인**

입찰 등록, 최고가 갱신, 마일리지 예약/해제 처리가 서로 다른 도메인에 걸쳐 있어 단순 저장 순서만으로는 데이터 정합성을 보장하기 어려웠습니다.

**해결 방법**

입찰 처리 흐름을 하나의 도메인 흐름으로 묶고, 현재 경매 상태와 최고 입찰 금액을 재검증한 뒤 입찰을 저장하도록 구성했습니다. 새 최고 입찰자가 확정되면 이전 최고 입찰자의 예약 마일리지를 해제하고 신규 입찰자의 마일리지를 예약하도록 처리했습니다.

**결과**

동시 입찰 상황에서도 최고 입찰자, 현재 최고가, 마일리지 예약 금액이 일관되게 유지되도록 개선했습니다. 금전성 데이터가 포함된 기능에서는 트랜잭션 경계와 상태 검증이 중요하다는 점을 경험했습니다.

---

### Slide 21. Troubleshooting 2 - Admin Access Control

**문제 상황**

관리자 메뉴와 버튼을 프론트에서 숨기더라도 사용자가 URL을 직접 입력하거나 API를 직접 호출하면 관리자 기능에 접근을 시도할 수 있었습니다.

**원인**

초기 화면 제어는 사용자 경험 측면의 처리였고, 실제 보안 경계는 서버에서 강제되어야 했습니다. 프론트 UI 제어만으로는 관리자 권한이 필요한 데이터와 액션을 보호할 수 없었습니다.

**해결 방법**

Spring Security 설정에서 `/api/admin/**` 경로는 관리자 권한을 가진 사용자만 접근 가능하도록 제한했습니다. 프론트에서는 권한별 메뉴와 버튼을 숨기되, 서버에서는 요청마다 권한을 재검증하도록 구성했습니다.

**결과**

권한 없는 사용자가 직접 관리자 URL에 접근하거나 API를 호출하더라도 서버에서 차단되도록 개선했습니다. UI 편의성과 서버 보안을 분리해야 한다는 점을 경험했습니다.

---

### Slide 22. Troubleshooting 3 - Auction Closing State

**문제 상황**

경매 종료 시간이 지났지만 서버 상태 갱신이 아직 반영되지 않은 순간에는 화면상으로 입찰 가능한 것처럼 보이거나, 사용자가 마감 직후 입찰을 시도할 수 있는 문제가 있었습니다.

**원인**

경매 상태는 서버의 `OPEN/CLOSED` 값과 클라이언트의 현재 시간 기준 마감 여부가 함께 고려되어야 했습니다. 서버 상태만 화면에 반영하면 마감 시간 직후의 짧은 구간에서 UI와 실제 입찰 가능 여부가 어긋날 수 있었습니다.

**해결 방법**

프론트에서는 `endAt`과 현재 시간을 기준으로 마감 여부를 계산해 버튼과 상태 배지를 즉시 비활성화했습니다. 동시에 서버 입찰 API에서도 경매 상태와 마감 시간을 재검증해 클라이언트 우회 요청을 차단했습니다. 마감 예상 시점에는 상세 정보와 입찰 내역을 다시 불러오도록 타이머를 구성했습니다.

**결과**

경매 마감 직후에도 사용자는 화면에서 일관된 상태를 확인할 수 있고, 실제 입찰 가능 여부는 서버에서 최종 검증되도록 개선했습니다. 시간 기반 도메인에서는 클라이언트 표시와 서버 검증을 함께 설계해야 한다는 점을 배웠습니다.

## 4-7. 회고

### Slide 23. Retrospective

**배운 점**

- 거래, 경매, 채팅, 마일리지처럼 서로 연결된 기능은 개별 CRUD보다 상태 전이 설계가 중요했습니다.
- 권한 제어는 프론트 버튼 숨김만으로 충분하지 않고 서버에서 반드시 강제해야 한다는 점을 체감했습니다.
- Hook dependency warning은 단순 경고가 아니라 데이터 로딩 구조를 점검하는 신호라는 점을 배웠습니다.

**아쉬운 점**

- 테스트 범위가 핵심 기능 중심으로 제한되어 전체 E2E 테스트는 부족했습니다.
- Redis 같은 외부 의존성이 있는 기능은 테스트 환경 분리가 더 필요했습니다.
- 실제 배포 환경 기준의 모니터링과 로그 설계는 추가로 보완할 수 있습니다.

**개선 방향**

- 경매 입찰 동시성 테스트 강화
- Redis 의존 기능 테스트 프로파일 분리
- 관리자 감사 로그 및 운영 통계 고도화
- 주요 사용자 흐름에 대한 E2E 테스트 추가
